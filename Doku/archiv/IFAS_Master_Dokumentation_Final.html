1. Vision & Problemstellung

Das Problem: Manuelle Erfassung (IST-Zustand)

Ineffizienter Prozess belastet Ressourcen
Täglich gehen verschiedenste Meldungen per E-Mail und Post ein. Anzeigen wie Mutterschutzmeldungen oder Pyroanzeigen werden dabei überwiegend manuell erfasst: Dokument öffnen → Betrieb suchen → Daten abtippen → Zuordnen → Speichern. 5-15 Minuten pro Fall. Bei 19.600 Anzeigen/Jahr = 2.287 Arbeitsstunden verschwendet für reine Dateneingabe.

Typische Probleme beim manuellen Prozess:

  • Zeitaufwendig: 5-15 Min pro Anzeige
  • Fehleranfällig: Tippfehler, falsche Zuordnungen, inkonsistente Erfassung
  • Nicht skalierbar: Bei steigenden Fallzahlen müsste Personal aufgestockt werden
  • Keine Datenqualität: Unterschiedliche Erfassungsstandards je nach Sachbearbeiter
  • Verzögerungen: Posteingänge bleiben liegen, falsche Weiterleitung

Die Vision: Arbeitsschutz 4.0

Vom Posteingang zur Anzeige in unter 2 Minuten
Das LASG digitalisiert die Arbeitsschutzverwaltung durch Künstliche Intelligenz. Eingehende Meldungen – von Mutterschutz über Pyrotechnik bis Sprengarbeiten – werden automatisch verarbeitet, kategorisiert und den zuständigen Sachbearbeitern zugeordnet. Ergebnis: 60-80% Zeitersparnis, höhere Datenqualität, schnellere Bearbeitung.

4. Intelligentes Sachbearbeiter-Routing

Suchfunktion über IFAS-API
Mittels API soll automatisch den zuständigen Sachbearbeiter basierend auf der Betriebsstätte ermittelt werden. Kein manuelles Routing mehr – der Posteingang landet direkt beim richtigen Kollegen oder wird an die Abteilungsleitung weitergeleitet, wenn kein Zuständiger hinterlegt ist.

Der 3-Stufen-Routing-Prozess

Stufe 1: Betriebsstätten-Suche (Fuzzy Matching)

API durchsucht IFAS nach Name, PLZ, Ort. Selbst bei Tippfehlern oder unvollständigen Angaben findet das System den richtigen Betrieb mit einem Confidence-Score.

GET /betriebsstaetten/search?name=RSA%20Testbetrieb&plz=22880

→ Response: {
    "betriebsstaetten_id": "BS-123456",
    "name": "RSA Testbetrieb GmbH",
    "match_score": 0.95  // 95% Übereinstimmung
}

Stufe 2: Zuständigkeits-Ermittlung

Falls Betrieb gefunden: API holt den zuständigen Sachbearbeiter aus der IFAS-Datenbank.

GET /betriebsstaetten/BS-123456/zustaendiger

→ Response: {
    "zustaendiger": {
        "mitarbeiter_id": "MA-98765",
        "name": "Schmidt, Thomas",
        "rolle": "Sachbearbeiter Mutterschutz",
        "email": "thomas.schmidt@lasg.landsh.de"
    },
    "vertretung": {...}
}

Stufe 3: Automatische Weiterleitung (3 Modi)

Modus A: Auto nach Betriebszuständigem

POST /posteingaenge/{id}/weiterleiten
{
  "modus": "auto_betriebszustaendig",
  "betriebsstaetten_id": "BS-123456"
}

→ System findet automatisch den Zuständigen

Modus B: Auto Fallback (bei unbekanntem Betrieb/Zuständigem)

{
  "modus": "auto_fallback",
  "grund": "Betrieb nicht in IFAS gefunden"
}

→ System leitet an Abteilungsleitung weiter

Modus C: Manuell (Nutzer wählt Empfänger)

{
  "modus": "manuell",
  "ziel_mitarbeiter_id": "MA-98765",
  "nachricht": "Bitte übernehmen"
}

Routing-Szenarien im Überblick

Situation System-Aktion Endpunkt
✅ Betrieb gefunden + Zuständiger bekannt Automatische Zuordnung zum Sachbearbeiter GET /betriebsstaetten/{id}/zustaendiger
⚠️ Betrieb gefunden + Kein Zuständiger Fallback an Abteilungsleitung GET /mitarbeiter/fallback
❌ Betrieb nicht gefunden Manuelle Weiterleitung erforderlich POST /posteingaenge/{id}/weiterleiten
📧 Allgemeine Anfrage ohne Betrieb Weiterleitung an Fachbereichsleiter POST /posteingaenge/{id}/weiterleiten

5. API-Endpunkte (Detaillierte Spezifikation)

Die API folgt REST-Prinzipien und ist vollständig in OpenAPI 3.0 dokumentiert. Hier die wichtigsten Endpunkte für Phase 1:

GET /arbeitskorb/posteingaenge
Core Feature

Beschreibung: Ruft Posteingänge aus dem IFAS-Arbeitskorb ab. Unterstützt Filterung nach Status, Eingangsart, Datum und Pagination.

Query-Parameter:

  • status - neu, in_bearbeitung, weitergeleitet, abgeschlossen
  • eingangsart - MUTTERSCHUTZ, PYROTECHNIK_FEUERWERKSK, etc.
  • von_datum / bis_datum - YYYY-MM-DD Format
  • limit (max: 200) / offset - Pagination

Response (200 OK):

{
  "success": true,
  "count": 3,
  "total": 15,
  "results": [
    {
      "posteingang_id": "POST-2026-00123",
      "eingangsdatum": "2026-01-15T08:30:00Z",
      "eingangsart": "MUTTERSCHUTZ",
      "betreff": "Mutterschutz-Meldung Maria Mustermann",
      "status": "neu",
      "absender": {
        "name": "RSA Testbetrieb",
        "email": "personal@rsa-testbetrieb.de"
      },
      "erkannte_daten": {
        "betrieb_name": "RSA Testbetrieb",
        "betrieb_plz": "22880",
        "confidence": 0.92
      },
      "anhaenge": [...]
    }
  ]
}
POST /anzeigen
Core Feature

Beschreibung: Erstellt eine neue Anzeige in IFAS. Validiert automatisch Pflichtfelder je nach Anzeigenart.

Request Body (Beispiel: Mutterschutz):

{
  "betriebsstaetten_id": "BS-123456",
  "anzeigenart": "MUTTERSCHUTZ",
  "anzeige_vom": "2026-01-15",
  "anzeige_aufgenommen_durch": "IFAS-Assist",
  "betreff": "Mutterschutz-Anzeige für Maria Mustermann",
  "felder": {
    "name_schwangere": "Maria Mustermann",
    "status": "schwanger",
    "beschaeftigungsort": "Wedel, Produktionshalle",
    "voraussichtlicher_entbindungstermin": "2026-08-15"
  },
  "posteingang_id": "POST-2026-00123"
}

Response (201 Created):

{
  "success": true,
  "message": "Anzeige erfolgreich erstellt",
  "data": {
    "anzeigen_id": "ANZ-2026-001234",
    "vorgangsnummer": "IZ 000 052 425",
    "aktenzeichen": "Vg. 123/2026-DU",
    "status": "erfasst"
  }
}
POST /anzeigen/{id}/schreiben
Kritisch

Beschreibung: Hängt ein Schreiben (Dokument, E-Mail, Nachtrag) an eine bestehende Anzeige an. Entspricht dem Karteireiter "Schreiben" in IFAS.

Wichtig: Nicht jeder Posteingang wird eine neue Anzeige! Änderungsmeldungen, Nachträge oder Rückfragen zu bestehenden Vorgängen werden als "Schreiben" angehängt, nicht als neue Anzeige erfasst.

Request Body:

{
  "posteingang_id": "POST-2026-00125",
  "schreiben_typ": "eingang",
  "datum": "2026-01-20",
  "betreff": "Änderungsmeldung - neuer Entbindungstermin",
  "bemerkung": "Entbindungstermin geändert auf 2026-08-20",
  "anhaenge_uebernehmen": true,
  "felder_aktualisieren": {
    "voraussichtlicher_entbindungstermin": "2026-08-20"
  }
}

Response (201 Created):

{
  "success": true,
  "message": "Schreiben erfolgreich angehängt",
  "data": {
    "schreiben_id": "SCH-2026-00042",
    "anzeigen_id": "ANZ-2026-001234",
    "felder_aktualisiert": true
  }
}
GET /betriebsstaetten/search
Smart Search

Beschreibung: Sucht Betriebsstätten mit Fuzzy-Matching. Findet Treffer auch bei Tippfehlern oder unvollständigen Angaben.

Query-Parameter:

  • name - Name des Betriebs
  • plz - Postleitzahl
  • ort - Ort
  • fuzzy - Fuzzy-Matching aktivieren (default: true)

Response:

{
  "success": true,
  "results": [
    {
      "betriebsstaetten_id": "BS-123456",
      "name": "RSA Testbetrieb GmbH",
      "adresse": {...},
      "match_score": 0.95,
      "aktiv": true
    }
  ]
}
GET /betriebsstaetten/{id}/zustaendiger
Smart Routing

Beschreibung: Ermittelt den zuständigen Sachbearbeiter für eine Betriebsstätte. Kern-Feature für automatisches Routing.

Response (200 OK):

6. Technische Anforderungen & Sicherheit

Entscheidung im Kick-off erforderlich: Der finale Technologie-Stack (Spring Boot vs. FastAPI vs. ASP.NET) wird im Workshop mit Kisters festgelegt. LASG empfiehlt: Technologieoffen, aber container-ready und gut wartbar.

Sicherheitsanforderungen (zwingend)

  • TLS 1.3: Verschlüsselte Übertragung (erzwungen, keine Ausnahmen)
  • JWT Auth: Token-basiert mit auto. Rotation (1h Laufzeit)
  • IP-Whitelist: Nur LASG-IP-Adressen erlaubt
  • Rate Limiting: Max. 100 Requests/Minute pro API-Key
  • Input-Validierung: Schutz gegen SQL-Injection, XSS, CSRF
  • Audit-Logging: Alle Zugriffe protokolliert (Zeitstempel, User, Aktion)
  • DSGVO: Separater AVV (Auftragsverarbeitungsvertrag)

Performance & Qualität

  • Antwortzeit GET: < 500ms (95. Perzentil)
  • Antwortzeit POST: < 2 Sekunden (95. Perzentil)
  • Skalierbarkeit: Min. 50 parallele Requests
  • Verfügbarkeit: > 99.5% (Mo-Fr, 7-18 Uhr)
  • Testabdeckung: > 80% Unit-Tests
  • Dokumentation: Vollständig (OpenAPI, Swagger UI, Postman)

7. Phase 2: IFAS-Assist (Interne Entwicklung)

Folgt nach erfolgreicher API-Fertigstellung
Der IFAS-Assist wird intern durch LASG SB 403 IT entwickelt, sobald die API aus Phase 1 produktionsreif ist. Die API ist die technische Voraussetzung für den Assistenten.

Was ist der IFAS-Assist?

Der IFAS-Assist ist eine webbasierte Anwendung, die Sachbearbeitern die Erfassung von Anzeigen durch KI-Unterstützung massiv erleichtert:

Backend (Python)

  • Automatischer Abruf neuer Posteingänge via API
  • KI-Datenextraktion (PDF OCR + NLP)
  • Automatische Betriebssuche (Fuzzy)
  • Smart Routing an Zuständige

Frontend (Web)

  • Übersichtliche Posteingangs-Liste
  • Extrahierte Daten zur Validierung
  • "Ein-Klick-Erfassung" nach Prüfung
  • Korrektur-Interface bei Fehlern

KI-Features des IFAS-Assist

PDF/E-Mail Extraktion

OCR + Natural Language Processing extrahiert Daten aus:

  • Ausgefüllten Formularen
  • Fließtexten
  • Tabellen
  • Unterschriften (Metadaten)

Auto-Kategorisierung

Machine Learning erkennt automatisch:

  • Anzeigenart (Mutterschutz, Pyro, etc.)
  • Betriebsname/-adresse
  • Datumsangaben
  • Personendaten

Validierung

System prüft automatisch:

  • Vollständigkeit Pflichtfelder
  • Plausibilität (Datumslogik)
  • Duplikate
  • Confidence-Score
